Sistema de Sincronização Incremental

Gerenciamento Inteligente de Precatórios

Visão Geral

Este sistema realiza sincronização incremental entre arquivos CSV dos tribunais (TJPR e TJSC) e o banco de dados PostgreSQL, mantendo os dados sempre atualizados sem duplicações ou perdas de informação.

Objetivos do Sistema

  • Sincronizar dados de precatórios de forma eficiente e segura
  • Preservar modificações manuais feitas no banco de dados
  • Identificar novos registros automaticamente
  • Atualizar registros existentes quando necessário
  • Marcar registros removidos como "não está na ordem"
  • Manter histórico completo de todas as alterações

Conceitos Fundamentais

Chave Única

Chave Única = (Organização, Tribunal, Ano Orçamentário, Prioridade)

Um precatório é identificado unicamente pela combinação desses quatro campos. Esta chave permite diferenciar precatórios parcelados ou com múltiplas prioridades, distribuídos por diferentes anos orçamentários.

Estados do Registro

Está na Ordem

Precatório presente no CSV atual

esta_na_ordem = true

Não Está na Ordem

Precatório foi removido do CSV

nao_esta_na_ordem = true

Possibilidades de Sincronização

Possibilidade 1: Novo Registro

Situação: Precatório está no CSV mas não está no banco de dados.

Ação: Inserir novo registro com status "está na ordem".

INSERT INTO precatorios (organizacao, ordem, precatorio, prioridade, tribunal, esta_na_ordem, ...) VALUES (..., true)

Possibilidade 2: Atualizar Registro Existente

Situação: Precatório está no CSV e no banco, mas os dados mudaram.

Verificação: Sistema verifica se há modificações manuais nos logs.

Decisão:

  • Se não há logs: Atualiza os dados do CSV
  • Se há logs: Preserva os dados existentes no banco
-- Com logs: PRESERVA dados -- Sem logs: ATUALIZA dados do CSV

Possibilidade 3: Registro Removido

Situação: Precatório está no banco mas não está mais no CSV.

Ação: Alterar status para "não está na ordem".

UPDATE precatorios SET esta_na_ordem = false, nao_esta_na_ordem = true WHERE ...

Fluxo Completo de Sincronização

1 Carregamento dos Dados Sistema carrega e processa os arquivos CSV dos tribunais TJPR e TJSC.
↓
2 Backup Preventivo Cria backup completo do estado atual do banco antes de qualquer modificação.
↓
3 Processamento dos Dados na Ordem Para cada registro presente no CSV:
  • Busca no banco pela chave única (organização, tribunal, ano orçamentário, prioridade)
  • Novo registro: Insere no banco
  • Registro existente sem mudanças: Mantém como está
  • Registro existente com mudanças e sem logs: Atualiza dados
  • Registro existente com mudanças e com logs: Preserva dados do banco
↓
4 Processamento dos Dados Fora da Ordem Para cada registro existente no banco:
  • Verifica se está presente no CSV atual
  • Se não está: Marca como "não está na ordem"
  • Atualiza status: esta_na_ordem = false, nao_esta_na_ordem = true
↓
5 Backup Pós-Sincronização Cria backup completo do novo estado do banco após todas as alterações.
↓
6 Confirmação e Finalização Confirma todas as transações e encerra a conexão com o banco de dados.

Lógica de Preservação

O sistema implementa uma proteção inteligente para modificações manuais:

  • Se um precatório possui registros na tabela de logs, significa que foi modificado manualmente
  • Neste caso, o sistema não atualiza os dados vindos do CSV
  • Os dados modificados manualmente são preservados no banco
  • Garante que o trabalho manual do usuário não seja sobrescrito

Nota sobre Tempo de Processamento

Primeira Sincronização (~5 minutos): Necessária apenas na primeira vez, quando o banco está vazio ou tem poucos dados. Insere todos os 84.406 registros.

Sincronizações Incrementais (~1-2 minutos): A partir da segunda execução, o processo é muito mais rápido, pois:

  • Apenas novos registros são inseridos (poucos)
  • Apenas registros com mudanças são atualizados (poucos)
  • Registros removidos apenas têm status alterado (atualização simples)
  • Tudo é processado em memória (sem consultas ao banco)
  • A maioria dos registros não precisa de nenhuma operação (já estão corretos)

Otimização: Sistema usa carregamento em memória (2 consultas ao invés de 168.000). Processamento incremental é extremamente eficiente.

Como Funciona a Otimização

# OTIMIZAÇÃO IMPLEMENTADA - Carrega TODOS os dados de uma vez (1 consulta) all_precatorios = db.fetch_all("SELECT * FROM precatorios") # Carrega TODOS os logs de uma vez (1 consulta) all_logs = db.fetch_all("SELECT * FROM precatorios_logs") # Cria dicionários em memória para busca O(1) precatorios_dict = {} for p in all_precatorios: key = (p['organizacao'], p['prioridade'], p['tribunal']) precatorios_dict[key] = p logs_dict = {} for l in all_logs: key = (l['organizacao'], l['prioridade'], l['tribunal']) logs_dict[key] = l # Processa tudo em memória (sem consultas ao banco) for data in csv_data: key = (data['organizacao'], data['prioridade'], data['tribunal']) existing = precatorios_dict.get(key) # O(1) lookup logs = logs_dict.get(key, []) # O(1) lookup # Processa sem consultar banco
